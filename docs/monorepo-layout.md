# Monorepo layout

fb2skill is a **mixed-language monorepo** with four packages under `packages/`. Three are Python (uv-workspace members); one is a TypeScript/React frontend (npm-managed). The split mirrors the package style of [NordicAgents/predicate](https://github.com/NordicAgents/predicate) (pnpm workspace) and matches PLC2Skill's own Maven multi-module structure (`lib/`, `cli/`, `rest-api/`).

```
fb2skills/
├── pyproject.toml                       # uv workspace root — no [project] table
├── uv.lock                              # single lockfile for all Python members
├── packages/
│   ├── fb2skill-core/                   # python — domain logic (no CLI, no HTTP)
│   ├── fb2skill-cli/                    # python — `fb2skill` console script
│   ├── fb2skill-rest/                   # python — FastAPI server + SPA static mount
│   └── fb2skill-web/                    # typescript — React/Vite SPA (npm-managed)
└── docs/
```

The root uv workspace excludes `fb2skill-web` because it has no `pyproject.toml`:

```toml
[tool.uv.workspace]
members = ["packages/*"]
exclude = ["packages/fb2skill-web"]
```

## Dependency graph

```
fb2skill-cli  ──▶ fb2skill-core            (python import)
fb2skill-rest ──▶ fb2skill-core            (python import)
fb2skill-web  ──▶ fb2skill-rest            (HTTP at runtime)
```

Strict DAG, no back-edges. The Python packages never depend on the web package; the web package never depends on Python packages at build time — only on the REST API at runtime.

## What lives in each package

### fb2skill-core
- Pure Python library. Dependencies: `lxml`, `Jinja2`, `rdflib`.
- Modules: `model`, `state_machine`, `fbt_parser`, `opcua_nodes`, `project`, `render`, `verify`, `config` (`RenderConfig` only).
- Bundled assets via `[tool.hatch.build.targets.wheel.force-include]`:
  - `templates/*.j2` — TTL Jinja templates
  - `data/CaSkMan_v4.3.0.ttl` — canonical ontology (served by REST `/ontology`)
- Tests live alongside under `tests/` with FESTO-dependent cases gated by `requires_festo`.
- Exemplar TTL fixtures in `tests/exemplars/{skLoad,skPush,skTransfer}.ttl` — graph-isomorphism reference for the render pipeline.

### fb2skill-cli
- Thin orchestration over core. Wraps `RenderConfig` in `CliConfig` (adds `project_root`, `out_dir`, `opcua_xml`, `verify`).
- Console script: `fb2skill` → `fb2skill_cli.cli:main`.

### fb2skill-rest
- FastAPI app factory in `app.py`; routers split by resource (`health`, `convert`, `skills`, `ontology`).
- Service layer (`services/conversion.py`) is the **only** place core is invoked; routers stay thin.
- Per-request lifecycle: extract uploaded zip to `tempfile.TemporaryDirectory()`, run discovery + render, return JSON, cleanup. Includes Zip-Slip guard.
- Settings via `pydantic-settings`, env-prefixed `FB2SKILL_*`. CORS allows `localhost:5173` (Vite dev) by default.
- **SPA hosting:** auto-detects `packages/fb2skill-web/dist/` and mounts it. `/assets/*` serves bundled JS/CSS; a catch-all falls back to `index.html` so React Router handles client-side routes. API routes (declared before the catch-all) take precedence.
- Dockerfile + docker-compose for containerized deployment; the build context is the repo root (workspace-aware `uv sync --package fb2skill-rest`).

### fb2skill-web
- React 18 + TypeScript + Vite 5 + Tailwind 4. Routing via `react-router-dom`. State-machine viz via `@xyflow/react`. Turtle syntax highlighting via `highlight.js` with an inline custom turtle language (~30 lines) — no extra language package needed.
- Four pages: `ConvertPage`, `DiscoverPage`, `OntologyPage`, `VisualizePage`. Shared components in `src/components/` (`Layout`, `TtlViewer`, `ZipUploader`).
- API client in `src/api/client.ts` mirrors the Pydantic schemas (`src/types/api.ts`). All fetches are same-origin — Vite proxies in dev, fb2skill-rest serves the built bundle in prod.
- `src/state_machine.ts` mirrors `fb2skill_core/state_machine.py` (STATES, COMMANDS, transition tables, layout coordinates). Kept in sync manually — if the Python tables change, mirror them here.

## Workspace mechanics

Root `pyproject.toml`:
```toml
[tool.uv.workspace]
members = ["packages/*"]
```

Each member's `pyproject.toml` declares its inter-package dependency:
```toml
dependencies = ["fb2skill-core"]
[tool.uv.sources]
fb2skill-core = { workspace = true }
```

Single `uv.lock` at the root resolves the entire graph.

## Config split rationale

The original single `Config` dataclass mixed two concerns:
1. **Render parameters** used by the templates (endpoint URL, base IRI, resource, namespace index, provenance labels).
2. **CLI-only parameters** never relevant to a REST request (`out_dir`, `verify`).

After the split:
- `fb2skill_core.config.RenderConfig` — render-only fields, plus free-form `project_label` / `source_label` for the header template's "Generated by fb2skill from …" lines. The CLI fills them with `str(project_root)` and `opcua_xml.name`; REST fills them with the uploaded zip's filename and the detected XML's basename.
- `fb2skill_cli.config.CliConfig` — CLI-only fields wrapping a `RenderConfig`.

## Open / future items

- **Auth on `/convert`** — none implemented; deploy behind a reverse proxy with auth if exposed publicly.
- **Ontology versioning** — `GET /ontology` currently returns the bundled v4.3.0; future revisions could expose `GET /ontology/v4.3.0` etc.
- **Upload-size enforcement** — `Settings.max_upload_mb` is wired but not enforced as middleware yet; rely on reverse-proxy `client_max_body_size` for now.
- **`verify.py` placement** — kept in `fb2skill-core` because both the CLI's `--verify` flag and REST's headline test reuse it. A future `fb2skill-testing` package could host it if a "no test deps in production" policy is adopted.
